# 模型解释完整性设计

## 问题

模型响应达到输出上限时，旧实现忽略 `finish_reason`，直接保存并展示半截文本。页面因此会出现未闭合的句子、括号或列表，用户无法区分完整结论与截断内容。

## 设计

- 首次输出上限提高到 2400 tokens，同时要求六个固定小节各不超过三条、全文优先控制在 1200–1800 个中文字符。
- 提示词要求模型在全部小节完成后输出隐藏完成标记。
- 后端同时校验完成标记和 `finish_reason`，不再把 `length` / `max_tokens` 响应视为成功。
- 仅在响应不完整时续写一次，续写上限 1200 tokens，并要求从断点继续、不得重复。
- 拼接成功后移除完成标记；二次响应仍不完整则返回错误，不保存、不展示残缺解释。
- `model_execution` 记录完成状态、结束原因、续写次数、完成 token 数与最终字符数，便于审计和成本分析。

## 验收

- 正常完整响应只调用模型一次。
- `finish_reason=length` 会触发一次续写并正确拼接。
- 缺少完成标记或续写仍截断时，研判失败且残缺文本不会进入报告。
- DeepSeek 与其他 OpenAI 兼容模型遵循相同契约。
